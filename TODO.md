# TODO: Remote-Client Netcode Bugs

This file tracks mods with confirmed multiplayer desync bugs affecting **remote clients**
(pure clients with no local authority — see `netcode.md` for the NGO host-authoritative
model). All of these mods work correctly for the host and in singleplayer; the bugs below
only manifest for players who joined someone else's game as a non-host client.

Generated by a per-mod audit of every mod in this repo except `ConstructToInventory`
(actively being fixed — see recent commits) and `AedenthornUtils`/shared code (not a mod).
Each mod below got its own investigative pass; mods not listed here were checked and found
clean (see "Audited clean" section at the bottom), with one exception noted under
"Not audited".

---

## AutoMine

`AutoMine/BepInExPlugin.cs` mines resources automatically and deposits them directly into
inventories/world objects using non-networked APIs.

- `BepInExPlugin.cs:191` — calls `Inventory.AddItem()` directly (plain `List<T>` mutation)
  instead of `InventoriesHandler.Instance.AddItemToInventory(...)`. On a remote client this
  item never replicates to the host or other players — it appears to succeed locally then
  desyncs/vanishes on the next inventory refresh from the server.
- `BepInExPlugin.cs:196` — same anti-pattern, second call site (different resource path).
- `BepInExPlugin.cs:198` — same anti-pattern, third call site.
- **Fix:** route all three call sites through `InventoriesHandler.Instance.AddItemToInventory(...)`
  with the async callback pattern (see `netcode.md` §3, and the QuickStore/CraftFromContainers
  fixes for a worked example of restructuring a synchronous call site into the async form).

## ChatCommands

`ChatCommands/BepInExPlugin.cs` implements admin/cheat-style chat commands that spawn items
and mutate inventories directly.

- `BepInExPlugin.cs:279` — direct `WorldObjectsHandler.Instance.CreateNewWorldObject(...)`
  call (local allocation only, no `NetworkObject.Spawn`). On a remote client this creates a
  world object only that client can see; it's never replicated to the host or other players.
- `BepInExPlugin.cs:283` — direct `Inventory.AddItem()`/`RemoveItem()` mutation, same
  non-networked-list problem as AutoMine above.
- `BepInExPlugin.cs:288` — another direct-mutation call site (likely a "give item" command).
- **Fix:** replace spawn calls with `WorldObjectsHandler.CreateAndInstantiateWorldObject(...)`
  or `CreateAndDropOnFloor(...)`; replace inventory mutations with
  `InventoriesHandler.Instance.AddItemToInventory(...)` / `RemoveItemFromInventory(...)`.
  Since these are chat commands, also confirm they're actually being *executed* on the host
  (via RPC) rather than just locally on whichever client typed the command — if commands
  currently run client-side-only, a remote client's command may not affect the shared world
  at all even after the API swap.

## SpawnObject

`SpawnObject/BepInExPlugin.cs` provides a UI to spawn arbitrary world objects.

- `BepInExPlugin.cs:232` — direct `WorldObjectsHandler.Instance.CreateNewWorldObject(...)`,
  same local-only allocation issue as ChatCommands.
- `BepInExPlugin.cs:242` — direct `Destroy()` call instead of
  `WorldObjectsHandler.Instance.DestroyWorldObject(...)` — destroys the object locally only;
  other clients/host still see it, and it may reappear/desync on next world-state sync.
  `Instance` in this line's context refers to a `WorldObjectsHandler` used above at line 232.
- `BepInExPlugin.cs:352-363` — a block that manually places/positions spawned objects and
  writes their state directly rather than going through the networked spawn+placement flow;
  needs to be folded into the same `CreateAndInstantiateWorldObject`/`CreateAndDropOnFloor`
  fix so placement is replicated too.
- **Fix:** replace spawn/destroy calls with the networked equivalents per `netcode.md` §3.
  This mod is a good candidate for the QuickStore-style "upfront snapshot + optimistic local
  state + async callback" restructuring since it's UI-driven (user clicks "spawn", expects
  immediate visual feedback, but the actual spawn must go through the host).

## StorageAnywhere

`StorageAnywhere/BepInExPlugin.cs` lets players deposit items into any nearby container from
a distance.

- `BepInExPlugin.cs:217-218` — container discovery uses
  `FindObjectsByType<InventoryAssociated>()` and reflects the private `_inventoryId` field.
  This is the anti-pattern documented in `netcode.md` §7: on a remote client, containers are
  represented by `InventoryAssociatedProxy`, not `InventoryAssociated` — this scan silently
  finds zero (or the wrong) containers for a remote client, even though it works fine for the
  host. This overturns the *old* `netcode.md` audit table entry for this mod, which had
  marked it "Not a Bug" — it does have a bug, just a discovery-scan failure rather than a
  mutation-unsafety issue.
- `BepInExPlugin.cs:260-310` — the deposit flow built on top of that broken discovery scan;
  even once discovery is fixed, verify the actual item-transfer call in this range uses
  `InventoriesHandler.Instance.AddItemToInventory(...)` / `NetcodeUtils.MoveItemBetweenInventories(...)`
  rather than direct list mutation.
- `BepInExPlugin.cs:263`, `273-274` — specific direct-mutation call sites within the deposit
  flow above.
- **Fix:** swap the discovery scan to use the shared `AedenthornUtils/InventoryAssociatedUtils.cs`
  helpers — `FindCandidates(predicate, logDiag)` plus `ResolveInventory(candidate, callback)` /
  `ResolveInventoryCached(...)` — exactly as already done for QuickStore and
  CraftFromContainers (see `netcode.md` §7 for the full writeup and the "sharper gotcha" about
  async resolution racing synchronous Harmony postfixes). Source-link
  `InventoryAssociatedUtils.cs` in `StorageAnywhere.csproj` the same way the other two fixed
  mods do. Then fix the deposit call sites to use the async inventory-transfer APIs.

## StorageCustomization

`StorageCustomization/BepInExPlugin.cs` lets players resize storage containers.

- `BepInExPlugin.cs:119-162` — the resize flow calls `inventory.SetSize(...)` directly and
  then `componentInParent.SetInventory(i)` to push the resized inventory back onto the
  container component. This is a **previously-undocumented** anti-pattern (not one of
  netcode.md's existing three) — the safe path is
  `InventoriesHandler.Instance.SetInventorySize(Inventory, int sizeOffset, Vector3 dropPosition, Action<bool> result)`,
  which dispatches `SetInventorySizeServerRpc` so the resize is authoritative and replicates.
  On a remote client, calling `SetSize`/`SetInventory` directly resizes the container only
  for that client — the host and other players still see the old size, and any items placed
  in the "new" slots are invisible/lost to everyone else.
  - Specific direct-mutation call sites within that range: lines `126`, `134`, `138`, `142`,
    `146`, `150`, `154`, `159` (each corresponds to a different size-increment button/path).
- **Fix:** replace every `SetSize`/`SetInventory` pair in this block with a single call to
  `InventoriesHandler.Instance.SetInventorySize(...)`, using the `Action<bool> result`
  callback to drive any UI feedback (following the same "optimistic UI, async confirm"
  shape used elsewhere). This should probably be documented as a fourth anti-pattern in
  `netcode.md` §3/§4 once fixed, since it's a distinct mistake from plain Add/Remove misuse.

## BeaconToggleMenu

`BeaconToggleMenu/BepInExPlugin.cs` adds a beacon-renaming feature to the beacon toggle menu.

- `BepInExPlugin.cs:~188` — the rename handler writes directly to
  `WorldObjectText.textContainer.text` (a raw TextMeshPro string) instead of going through a
  networked rename path. Per QuickStore's own code comments and the decompiled IL for
  `WorldObject.SetText` (`reference/decompiled-il/WorldObject.il:844-853`), this text is
  actually **NetworkVariable-backed replicated state** — writing to the TMP component
  directly changes what the renaming client sees, but the host and every other client keep
  the old name, and the local-only change will likely be stomped on the next sync from the
  server.
- `BepInExPlugin.cs:~255` — a second call site with the same direct-write pattern (possibly
  a "reset to default name" path).
- **Fix:** find and use whatever networked rename entry point `WorldObject`/`WorldObjectText`
  actually exposes (check for a `SetText`-equivalent that dispatches an RPC — decompile
  `WorldObject.SetText` fully if `netcode.md`/`reference/decompiled-il/` doesn't already
  cover the full call chain) rather than writing the TMP text component directly.
  *(Note: this finding came from a subagent run flagged with a "safety classifier
  unavailable" caveat — the evidence, including the IL cross-check, looks solid, but give
  this one a quick manual re-read of the cited lines before treating the fix as fully
  specified.)*

## CustomWeatherEvents (different bug class — config-divergence risk)

`CustomWeatherEvents/BepInExPlugin.cs` and `WeatherData.cs` do **not** have the standard
direct-mutation bug — no unsafe `Inventory`/`WorldObjectsHandler` calls were found. Instead
there's a subtler, lower-confidence desync risk worth tracking separately:

- `BepInExPlugin.cs:73-139` (weather list construction) — the active weather event is synced
  across clients as a **NetworkVariable index** into a weather-event list that each client
  builds locally from its own config/data (`BepInExPlugin.cs:130`, `WeatherData.cs`).
- `BepInExPlugin.cs:194-213` (~`208`) — the index is read back and used to look up the
  "current" weather event from that same locally-built list.
- **The risk:** this only desyncs if a given client's local weather config differs from the
  host's (e.g. stale config file, different BepInEx config values, or a modified/reordered
  weather list) — in that case the same NetworkVariable index resolves to a *different*
  weather event on that client than on the host. This is not a bug in the netcode sense (no
  unsafe API misuse) but a **data-integrity assumption**: the sync-by-index scheme silently
  requires every client's local list to be identical and identically ordered to the host's.
  *(Note: this finding also came from a subagent run flagged with a "safety classifier
  unavailable" caveat — worth a manual skim before acting, though the reasoning is sound on
  its face.)*
- **Fix (lower priority than the sections above):** either sync the weather event by a
  stable string/enum ID instead of a list index (removes the ordering/identity assumption
  entirely), or add a startup check that validates each client's weather list matches the
  host's (count + order) and logs/warns on mismatch. Not urgent unless players are known to
  run mismatched configs.
